# DR-002: チャンク分割方式による日誌生成

## 背景

大規模セッション（100KB超のタイムライン）で、単一のclaude呼び出しだと内容が平坦化（「XXした。XXだった。」の羅列）してしまう問題があった。

## 不採用: Agent丸投げ方式

- claude -p --allowedTools Agent でサブエージェントに丸投げ
- 問題1: 130KBの会話テキストがサブエージェントにも丸ごと渡されコスト爆発（$2.78/件）
- 問題2: 出力が不安定（空結果になることも。164KB/97ターンのセッションで$1.80消費して空結果）
- --allowedTools はデフォルトツールに追加する形式（排他ではない）

## 採用: チャンク分割方式

タイムラインを統計分析 → 自然な境界で分割 → 各チャンクを並列でclaudeに渡す → メインが合成

### 分割ロジック

- 時間ギャップ + 日付境界 + サイズ制約のヒューリスティック
- AI判断不要、十分妥当な分割ができる
- 実装: src/lib/chunker.ts (csa-timeline-analyzer.ts から移植)

### 閾値設計

| パラメータ    | 値     | 根拠                                                         |
| ------------- | ------ | ------------------------------------------------------------ |
| maxChunkBytes | 35,000 | 実験で良好な品質。各チャンクが集中して深く書ける適切なサイズ |
| minChunkBytes | 8,192  | 小さすぎるチャンクの防止                                     |
| maxChunks     | 8      | コスト上限の制御                                             |

### 実験結果

kuu-review (164KB, 5日間97ターン):

|        | ベースライン | チャンク分割版                         |
| ------ | ------------ | -------------------------------------- |
| サイズ | 8.2KB        | 35.3KB (4.3倍)                         |
| コスト | $0.60        | $1.92 (chunks $1.38 + synthesis $0.54) |
| 品質   | 概要的       | エピソードを深く掘り下げ、密度が高い   |

kuu-main (127KB, 4時間10ターン):

|        | ベースライン | チャンク分割版             |
| ------ | ------------ | -------------------------- |
| サイズ | 4.8KB        | 17.4KB (3.6倍)             |
| コスト | ~$0.60       | $0.94                      |
| 品質   | 概要的       | 各フェーズの思考過程が深い |

### 品質面の発見

- チャンク分割で各区間に集中するため、平坦化を防げる
- Thinking の内容（英語の思考ログ）が日本語日誌に自然に翻訳される価値がある
- 同じプロジェクト・同じ週末でもワークスペース（review vs main）で日誌の性格がまるで違う
  - review: 「人を見ている」観察者視点
  - main: 「コードと戦っている」実装者視点

## プロンプト設計

- 合成プロンプト: セクションを並べ、タイトルとまとめを追加
